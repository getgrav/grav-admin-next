# i18n in admin-next (Admin2)

Admin-next translations are sourced from Grav's normal language pipeline:
every plugin's `languages/<lang>.yaml` is recursively merged by Grav core,
flattened by `Languages::flattenByLang()`, and served to the client by
`GET /api/v1/translations/{lang}`. No special build step or registration is
required.

What admin-next adds on top of that is a **lookup convention** that lets the
same plugin keep working on Grav 1 while opting in to richer message formatting
on Grav 2.

## The `ICU.` namespace

Admin-next looks up every key in two places, in order:

1. `ICU.<key>` â€” passed through [ICU MessageFormat][icu]
   (placeholders, plurals, select cases, number/date formatting).
2. `<key>` â€” returned raw, exactly as it appears in the YAML.
   Used as a fallback for legacy strings.

If neither is found, admin-next humanises the key (`ADMIN_NEXT.SOME_FIELD`
â†’ `"Some Field"`) so missing translations are still readable.

The contract is *namespace-based, not content-based*. A value goes through
the formatter only if its key starts with `ICU.`. Values without that prefix
are never reformatted, even if they happen to contain `{...}`.

## Shape of a plugin language file

For a plugin that only targets Grav 2 / admin-next, just put everything under
`ICU:`:

```yaml
ICU:
  PLUGIN_MYPLUGIN:
    TITLE: "My Plugin"
    GREETING: "Hello, {name}!"
    ITEMS_FOUND: "{n, plural, =0{No items} one{# item} other{# items}}"
    EDIT_LABEL: "{type, select, page{Edit page} post{Edit post} other{Edit item}}"
```

For a plugin that needs to support **both** Grav 1 / classic admin **and**
Grav 2 / admin-next from the same release, ship two parallel blocks:

```yaml
# Grav 1 / classic admin reads only this section.
PLUGIN_MYPLUGIN:
  TITLE: "My Plugin"
  GREETING: "Hello"
  ITEM_FOUND: "1 item found"
  ITEMS_FOUND: "items found"

# Admin-next prefers this section when present; Grav 1 ignores it entirely.
ICU:
  PLUGIN_MYPLUGIN:
    TITLE: "My Plugin"
    GREETING: "Hello, {name}!"
    ITEMS_FOUND: "{n, plural, =0{No items} one{# item} other{# items}}"
```

PHP/Twig code paths in your plugin keep referencing `PLUGIN_MYPLUGIN.GREETING`
and continue to work on both versions. Admin-next code paths (Svelte components
or web-component bundles) call `t('PLUGIN_MYPLUGIN.GREETING', { name })` and
get the ICU version on Grav 2, with automatic fallback to the flat string if
a translator hasn't filled in the `ICU.` block yet.

## Calling translations from admin-next

Inside Svelte components shipped in admin-next:

```ts
import { i18n } from '$lib/stores/i18n.svelte';

const label = i18n.t('PLUGIN_MYPLUGIN.GREETING', { name: user.name });
const exists = i18n.has('PLUGIN_MYPLUGIN.GREETING');
```

Inside a plugin web-component bundle (e.g. `editor-pro`, `ai-pro`) that is
loaded into the page but isn't built against the admin-next runtime, use the
read-only global bridge:

```js
const { t, has, locale, subscribe } = window.__GRAV_I18N;

const label = t('PLUGIN_EDITOR_PRO.TOOLBAR_BOLD');
const items = t('PLUGIN_MYPLUGIN.ITEMS_FOUND', { n: count });

// React to locale changes (e.g. user switches admin language)
const unsubscribe = subscribe((newLocale) => {
  rerender();
});
```

The global is installed during admin-next bootstrap and is not configurable â€”
it is a stable read-only surface for plugin bundles to depend on.

## ICU MessageFormat in 60 seconds

```
{name}                                 â†’ simple placeholder
{n, plural, one{# minute} other{# minutes}}
{type, select, page{Page} post{Post} other{Item}}
{n, number, percent}
{when, date, short}
```

Use `=0`, `=1` etc. for exact matches before plural categories:

```
{n, plural, =0{No results} one{1 result} other{# results}}
```

CLDR plural categories (`zero`, `one`, `two`, `few`, `many`, `other`) are
applied per locale automatically â€” Polish, Czech, Russian, Arabic etc. get
the right form without per-language code. Always include `other` as the
catch-all.

Translators can edit ICU strings in any tooling that understands the format
(Crowdin, Lokalise, Phrase, etc.). For hand-edited YAML, prefer the simpler
forms; reserve plural/select for the cases where English-grammar composition
actually breaks down.

## Migration tips

- For new admin-next-only strings: put them under `ICU.` from day one, even if
  they have no placeholders. Cost is zero and you avoid migrating later when a
  placeholder is added.
- For existing plugin strings: leave them where they are. Add an `ICU:` block
  only when you actually need formatting capability or want admin-next plurals.
- Don't auto-detect: the format is opt-in by namespace, not by inspecting the
  value. Keep the contract obvious.

## Auditing translations

Three small tools live in `scripts/` to help find translation gaps:

### `scripts/i18n-audit.mjs` â€” static reference scan

Walks `src/**/*.{svelte,ts}` for literal `i18n.t('KEY')` / `i18n.tMaybe('KEY')` /
`__GRAV_I18N.t('KEY')` calls, compares against the merged dictionary loaded
from `grav-plugin-admin/languages/en.yaml` + `grav-plugin-api/languages/en.yaml`
+ `grav-plugin-admin2/languages/en.yaml`, and reports:

- **gaps** â€” keys referenced in code but undefined anywhere (will humanize)
- **orphans** â€” `ICU.ADMIN_NEXT.*` keys defined but never literally referenced
  (note: many "orphans" may be referenced dynamically via `i18n.t(variable)` â€”
  treat the orphan list as candidates for review, not for deletion)
- **tMaybe-only** â€” keys only seen via `tMaybe`, where the input might be a
  plain string, not a key

```bash
node scripts/i18n-audit.mjs                    # report
node scripts/i18n-audit.mjs --json             # machine-readable
node scripts/i18n-audit.mjs --api https://...  # pull dict from a running API
node scripts/i18n-audit.mjs --update           # append @@TODO hints to admin2/languages/en.yaml
```

### `scripts/i18n-strings.mjs` â€” hardcoded string scan

Walks the source tree for English text that bypasses `i18n.t` entirely:

- text nodes between Svelte tags
- string literals in `placeholder=`, `aria-label=`, `title=`, `alt=` attributes
- string literal first arguments to `toast.success` / `toast.error` / `toast`

Skips obvious non-UI text (CSS classes, identifiers, file paths, numbers) and
the `routes/testing/` dev playground.

```bash
node scripts/i18n-strings.mjs                   # full report
node scripts/i18n-strings.mjs --dir lib/components  # scope to one subdir
node scripts/i18n-strings.mjs --json            # machine-readable
node scripts/i18n-strings.mjs --max 50          # cap findings per file
```

### `bin/plugin devtools i18n-blueprints` â€” blueprint YAML scan (any plugin)

Lives in [grav-plugin-devtools](https://github.com/getgrav/grav-plugin-devtools)
and ships as a Symfony Console command, since the audit is universally useful
for any plugin/theme author rather than admin-next-specific.

Walks `blueprints.yaml` and `blueprints/**/*.yaml` (also `admin/blueprints/`)
under each given plugin or theme directory. For every value of `label`,
`title`, `description`, `help`, `placeholder`, or `hint`, classifies it as a
translation key or hardcoded English. Reports per-plugin counts and a
hardcoded ratio.

```bash
bin/plugin devtools i18n-blueprints                          # current Grav install
bin/plugin devtools i18n-blueprints user/plugins/foo         # single plugin/theme
bin/plugin devtools i18n-blueprints user/plugins             # plugins dir only
bin/plugin devtools i18n-blueprints --details                # list every hardcoded value
bin/plugin devtools i18n-blueprints --threshold 0.5          # only show plugins â‰Ą50% hardcoded
```

With `--fix`, the command **rewrites the blueprint YAML in place**, replacing
each hardcoded value with a generated `PLUGIN_FOO.STRING_KEY` reference, and
emits the matching `KEY: "Original string"` entries to stdout for pasting
into the plugin's `languages/en.yaml`. Add `--dry-run` to preview, `--icu` to
emit keys under the `ICU.` block instead.

```bash
bin/plugin devtools i18n-blueprints user/plugins/foo --fix --dry-run
bin/plugin devtools i18n-blueprints user/plugins/foo --fix
bin/plugin devtools i18n-blueprints user/plugins/foo --fix --icu
```

The plugin prefix (`PLUGIN_FOO`) is read from the plugin's existing
`languages/en.yaml` if present (preserves the author's chosen convention),
otherwise derived from the directory name. Existing keys in the lang file
are reused â€” re-running `--fix` after editing translations is safe.

### Runtime humanize tracker

Admin-next can also log translation key misses at runtime â€” useful for catching
keys used only in dynamic code paths that the static scan misses.

In the browser console:

```js
__GRAV_I18N_DEBUG.enable();        // turns it on (persists in localStorage)
location.reload();                 // pick it up across all components

// click around the admin, then:
__GRAV_I18N_DEBUG.report();        // grouped console output
__GRAV_I18N_DEBUG.misses();        // raw array
__GRAV_I18N_DEBUG.yaml();          // YAML snippet of @@TODO entries
__GRAV_I18N_DEBUG.disable();       // turns it off
```

While debug is on, any humanize fallback is wrapped in unicode brackets so
gaps are visible directly in the rendered UI: `âź¦Some Fieldâź§` instead of
`Some Field`. You can also enable it via `?i18n-debug=1` on the URL.
